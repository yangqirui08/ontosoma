# OntoSoma · 愿景与哲学

> 本文是 OntoSoma 的"为什么"与"是什么"。技术落地见 [architecture.md](architecture.md);已锁定的决策见 [decisions.md](decisions.md)。

---

## 1. 问题:AI 开发的四个结构性痛点

1. **随机性** —— AI 写代码会跑偏,同一需求每次产出不同,无法收敛到设计意图。
2. **漂移** —— 写完之后,代码跟当初的设计/spec 对不上;时间越久,文档越像考古。
3. **重复理解** —— 每个会话 AI 都要从零重新"理解"代码库,而且每次理解都偏一点点。
4. **割裂** —— 设计文档、代码、运行时是三套独立的东西,谁也不保证跟谁一致。

这四个痛点的根因是**同一个**:系统的"真相"被复制成了多份(脑子里的设计、代码、文档、线上实际),而副本之间没有强制对账的机制。

---

## 2. 核心思想:一个本体,三个投影,只管理"差"

OntoSoma 的第一性原理:

> 把「设计意图 / 源代码 / 运行时」从"三份会互相漂移的副本",压成**同一个本体的三个投影**。然后整个方法论只做一件事——**管理投影之间的差**。

漂移不再靠纪律去防,而是**结构上不可能**,因为它们本来就是一个东西的不同看法。

这就是为什么我们叫它"本体"(ontology / 本体 / noumenon):

> **合格的本体,天然含"零漂移"和"唯一真源"两层含义。** 一个会跟代码对不上的"本体",根本不是本体,只是又一份会过期的文档。

---

## 3. 四根支柱

### 支柱一 —— 三位一体,零漂移(闭环)

系统代码、本体、本体展示是一体的:**系统代码就是本体代码,本体代码就是本体展示**。

- **代码是唯一真源**:本体的属性从代码**派生**(不是人手写声明),规则闸保证派生结果与代码一致,不一致就在启动/CI 时拦下。
- **关系/范畴可人工声明**,但锚定真实代码(有代码锚 = 软引用但锚定真实解析;外键 = 方向 + 校验)。
- **闭环对账**:这其实是一个控制系统——
  - 设计态本体 = 设定值(setpoint)
  - 实际(as-built)本体 = 反馈值
  - 两者的语义 diff = **误差信号**
  - 实现引擎(superpowers 等)= 执行机构;AI 的随机性 = 系统噪声

  它能"系统性"压住随机,不是让 AI 不犯错,而是**让任何偏离设计的产出无处藏身、必然显形**。

- **对象状态机**(闸据此区分"合法的未实现"与"非法的漂移"):
  - `intended` —— 已在设计态画出,尚未实现(合法的 design delta,放行但记账)
  - `built` —— 设计态 = 代码态,已对齐
  - `drifted` —— 代码态 ≠ 设计态(漂移,拦)
  - `orphan` —— 代码里有、设计态没有(野生,显形)

  开发的全部工作 = 把一切推到 `built`,并让 `drifted / orphan` 永远 = 0。

### 支柱二 —— 本体即文档(时间轴)

不再需要独立的、会过期的文档(包括 openspec)。

- **一切记录钉在本体上**:某个对象"为什么这么设计"、"何时变的"、决策与权衡,都作为批注贴在该对象、该时间帧上。文档锚本体,本体锚代码,**文档漂移结构性消失**。
- **时间轴 = 本体从 3D(结构)升到 4D(结构 × 时间)**。git 历史不是文本 diff 流,而是**本体形变的逐帧动画**。
  - 不是每个 commit 都是一帧——只有"让派生本体真的变了"的 commit 才算一帧,否则被噪声淹没。
  - 帧间 diff 是**语义的**(新增了能力 X、某关系变了),不是行 diff。
- 拖动时间轴,即可回放系统每个时间点的状态——而这段历史是**自动从代码长出来的**,不是人手写的。

### 支柱三 —— 低心智负担(镜头)

不是甩一张铺面而来的全景结构图,而是:**一个本体,N 个镜头**。

镜头 = **(高度 × 关注面 × 时间 × 焦点对象)**:

- **高度(altitude)**:系统骨架 → 模块 → 文件 → 符号,像地图的语义缩放。
- **关注面(concern)**:业务动作流 / 技术依赖 / 数据流 / 身份权限 / 运行时——像地图图层叠在同一张图上。
- **时间**:任一历史帧。
- **焦点**:任一对象为中心。

原则:**我只问一个具体问题,系统只给我这一片**。要警惕镜头本身变成新的心智负担——给一小撮"规范视角"做默认,外加"选中对象→选高度→自动出镜头"的动作,而不是无限可配。

### 支柱四 —— 运行态实时体现

把本体从"两态"扩成"三态":**设计态 / 代码态(静态 as-built)/ 运行态(as-running)**。

- 三态两两对账,长出新洞察:
  - **设计 ↔ 代码**:编译期闸(规则闸)。
  - **代码 ↔ 运行**:可观测性闸——代码登记了 tool 但运行时从没被调 → 死代码显形;运行时有个 MCP 在跑但本体里没有 → 影子组件显形;声明的交互运行时从没发生 → 关系是假的。
- **harness 也本体化**:MCP 注册在哪、skill 挂哪、tool 谁能调、cron/hook——都是对象、都进本体、都受三态对账。最终能在一张图上看到"这个 skill 调了哪个 MCP 的哪个 tool,那 tool 背后是哪段代码,那段代码上次运行是什么时候"。

这一步把**静态分析 + 可观测性 + 设计意图焊成一个模型**——这是 OntoSoma 区别于一切 spec 工具的根本:**spec 工具是标本,OntoSoma 是活体。**

## 3.5 展示层第一原则:对人友好 × 对AI准确

本体展示层(本体即文档、镜头、时间轴——一切给人和 AI 看的东西)只有一条根原则:

> **每一处可视展示,必须同时满足「对人理解友好」+「对AI准确」。** 两条都是硬约束,不是取舍。

- **对人理解友好**:易懂、低心智负担、用观看者的母语与心智模型。观看者是中国人 → 中文 + 承载概念的传统文化(阴阳静动、天地人、照镜…)。
- **对AI准确**:无歧义、可机读、忠实于代码真源;绝不为了"好懂"而丢失或扭曲真相。

二者**能同时成立**,是因为 AI 同样能准确理解中文与这些概念——所以"中文/传统文化"两轴都满足,是净赢。反过来,英文只保留在"翻译反而损害准确"处(机读 key、源码里的真实标识符)。**可见连"中文优先"都是从这条根原则推导出来的,而非独立规定。**

(完整落地约定见 [brand.md「展示层语言与文化约定」](brand.md);决策见 [decisions.md D7](decisions.md)。)

## 4. 全生命周期:不止开发,是开发 / 运维 / 迭代的共同基座

本体不是开发完就丢的脚手架,它是整个生命周期共读共写的**同一个活模型**:

- **共享世界模型**:开发 AI、运维 AI、迭代 AI 读的是同一个、且**此刻为真**的模型——不是 6 个月前的文档。AI 凌晨做运维时,看到的是系统现在真实的样子。
- **每个动作自动成帧**:运维改了什么、迭代加了什么,本身就是一次本体变更 = 时间轴上一帧。系统**自动积累自己的历史**。
- **对账持续生效**:运维洞察(死代码、影子组件、假关系)是三态对账白送的副产品。

这是我们认为的:**未来 AI 应用开发 / 运维 / 迭代的最终态。**

## 5. 与 openspec / superpowers 的定位

| | 解决什么 | 形态 | 局限 |
|---|---|---|---|
| superpowers | 怎么做(流程/思维) | Markdown skills | 不管"做出来跟设计是否一致" |
| openspec | 做之前说清楚 | 独立 spec 文件 | spec 与代码分离,会漂移;静态 |
| **OntoSoma** | 说清楚→做→验证一致→可追溯演化→运行时仍为真 | 从代码派生的活本体 + 闭环对账 | —— |

OntoSoma 不取代它们:**借鉴**其分发与跨工具适配的成熟做法,**多出**闭环对账与三态统一。

---

## 6. 杀招

两条:

1. **闭环(支柱一)** —— 治"AI 写跑偏"。
2. **三态对账(支柱四)** —— 治"写完跟现实又对不上"。

openspec 只解决"写之前说清楚";OntoSoma 把整条链路焊成一个不漏气的环,且本体是活的。
